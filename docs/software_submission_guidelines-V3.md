## 1. מבוא — המתכנת המקצועי בעידן הבינה המלאכותית

### 1.1 מהו מתכנת מקצועי?

מתכנת מקצועי (Professional Software Engineer) הוא לא רק מי שיודע לכתוב קוד, אלא מי שמבין את מלוא מחזור החיים של תוכנה, יודע לתכנן, לתעד, לבדוק ולתחזק מערכות מורכבות. מתכנת מקצועי עומד בסטנדרטים גבוהים של איכות קוד, מקפיד על שיטות עבודה מוכחות (best practices), ומסוגל לעבוד בצוות תוך שמירה על עקביות, תקשורת ברורה ואחריות.

המאפיינים המרכזיים של מתכנת מקצועי כוללים:

- **חשיבה מערכתית** — היכולת לראות את התמונה הגדולה ולהבין כיצד כל רכיב משתלב במערכת כולה
- **תכנון לפני ביצוע** — כתיבת מסמכי דרישות, ארכיטקטורה ותכנון לפני שורת קוד ראשונה
- **איכות ללא פשרות** — הקפדה על כתיבת קוד נקי, מתועד, מכוסה בבדיקות ובטוח
- **למידה מתמדת** — עדכון ידע בטכנולוגיות, כלים ומתודולוגיות חדשות
- **תקשורת אפקטיבית** — יכולת לתאר פתרונות טכניים לקהלים מגוונים

### 1.2 עבודת צוות בפיתוח תוכנה

פיתוח תוכנה מקצועי מתבצע כמעט תמיד בצוות. עבודת צוות אפקטיבית מבוססת על מספר עקרונות מפתח:

**תפקידים ואחריויות** — צוות פיתוח טיפוסי כולל ארכיטקט תוכנה (Software Architect) האחראי על תכנון המערכת, מפתחים (Developers) המיישמים את הקוד, מהנדס בדיקות (QA Engineer) המוודא את איכות המוצר, מנהל מוצר (Product Manager) המגדיר את הדרישות, ולעיתים גם מפתח DevOps האחראי על תשתיות ותהליכי פריסה.

**תהליכי עבודה** — צוותים מקצועיים עובדים עם מתודולוגיות כגון Scrum, Kanban או SAFe, הכוללות ישיבות יומיות (daily standup), סקירות ספרינט (sprint reviews), רטרוספקטיבות, וסקירות קוד (code reviews) הנדרשות באמצעות Pull Requests.

**סטנדרטים משותפים** — הצוות מגדיר ומקפיד על תקני קוד (coding standards), מבנה פרויקט אחיד, ותהליכי בדיקה ופריסה אוטומטיים (CI/CD).

### 1.3 מחזור חיי פרויקט תוכנה

כל פרויקט תוכנה מקצועי עובר מחזור חיים מוגדר (Software Development Life Cycle – SDLC):

1. **הגדרת דרישות** — כתיבת מסמך דרישות מוצר (PRD) מפורט
2. **תכנון וארכיטקטורה** — עיצוב המערכת, כתיבת מסמכי תכנון (PLAN) וקביעת אבני דרך (TODO)
3. **פיתוח** — כתיבת קוד לפי התכנון, בגישת TDD (Test-Driven Development)
4. **בדיקות** — בדיקות יחידה, אינטגרציה ומערכת
5. **פריסה** — הפצת התוכנה לסביבת הייצור
6. **תחזוקה ושיפור** — תיקון באגים, הוספת תכונות ושדרוגים

### 1.4 המהפכה: בינה מלאכותית וקידוד בהנחיה

בעידן הנוכחי, הבינה המלאכותית (AI) וסוכני AI משנים באופן מהותי את אופן פיתוח התוכנה. טכנולוגיית **קידוד בהנחיה** (Vibe Coding) — בה מתכנת מנחה סוכני AI ליצור, לבדוק ולשפר קוד — מאפשרת למתכנת בודד לתפקד כארכיטקט תוכנה בכיר (Senior Software Architect) המתזמר (orchestrate) מספר סוכני AI במקביל.

**הכפלת כושר הייצור:** מתכנת העובד עם סוכני AI ומשתמש בשיטת קידוד בהנחיה יכול לייצר בפרק זמן נתון פי 16 יותר שורות קוד איכותיות בהשוואה לכתיבה ידנית ללא AI. המשמעות היא שכל מתכנת, ללא קשר לוותק או ניסיון קודם, יכול להפוך למתכנת מקצועי ברמה גבוהה — בתנאי שהוא מקפיד על הכללים והתהליכים המוגדרים במסמך זה.

**הכלל הראשון והחשוב ביותר:** כדי לנצל את מלוא הפוטנציאל של סוכני AI, חובה להגדיר דרישות ברורות ומפורטות. ללא מסמכי דרישות, תכנון וארכיטקטורה — סוכני AI ייצרו קוד שאולי יעבוד, אבל לא יעמוד בסטנדרטים מקצועיים. לכן, **הכלל הראשון של קידוד מקצועי עם AI הוא להגדיר ולדרוש תיעוד מלא לפני כל שורת קוד**.

המסמך שלפניכם מגדיר את כל הדרישות, התקנים והתהליכים שיש לעמוד בהם כדי ליצור תוכנה מקצועית ברמה הגבוהה ביותר. עקבו אחר ההנחיות האלו, ותוכלו לנצל את מלוא הכוח של סוכני AI ולהפוך לארכיטקטים שמתזמר צוות של סוכנים — וייצר תוצאות ברמה מקצועית שלא היו אפשריות בעבר.

## 2. מבנה פרויקט חובה ומסמכי תיעוד

כל פרויקט תוכנה מקצועי **חייב** לכלול מבנה תיקיות ומסמכי תיעוד מינימליים. ללא מסמכים אלו, הפרויקט לא ייחשב כעומד בדרישות המינימום.

### 2.1 קובץ README.md בשורש הפרויקט

כל פרויקט **חייב** לכלול קובץ `README.md` בתיקיית השורש (root directory). קובץ זה משמש כמדריך למשתמש ברמת user manual מלא [1], [2] וכולל:

- **הוראות התקנה** (Installation Instructions) — דרישות מערכת, התקנה שלב-אחר-שלב, הגדרת משתני סביבה ופתרון בעיות נפוצות
- **הוראות הפעלה** (Usage Instructions) — הרצה במצבים שונים, דגלים ואפשרויות, CLI/GUI ו-workflow טיפוסי
- **דוגמאות והדגמות** — דוגמאות קוד, צילומי מסך, תרחישי שימוש נפוצים
- **מדריך תצורה** (Configuration Guide) — הסבר על קבצי קונפיגורציה, פרמטרים והשפעתם
- **הנחיות תרומת קוד** (Contribution Guidelines) — תקני קוד וסגנון
- **רישיון וייחוס** (License & Credits) — רישיון שימוש וייחוס לספריות צד שלישי

### 2.2 תיקיית /docs — מסמכי תיעוד חובה

כל פרויקט **חייב** לכלול תיקיית `/docs` בשורש הפרויקט, המכילה לפחות את המסמכים הבאים:

1. **מסמך דרישות מוצר** — `docs/PRD.md` (Product Requirements Document)

   מסמך ה-PRD הוא המסמך המרכזי המגדיר את מטרת הפרויקט ואת הדרישות ממנו [3], [4], [5]. המסמך כולל:

   - סקירת הפרויקט והקשר, תיאור בעיית המשתמש, ניתוח שוק וזיהוי קהל יעד
   - יעדים מדידים, מדדי KPI וקריטריוני קבלה (acceptance criteria)
   - דרישות פונקציונליות ולא-פונקציונליות, user stories ותרחישי שימוש
   - הנחות, תלויות, מגבלות ופריטים מחוץ לתחום
   - ציר זמן ואבני דרך עם משלוחים צפויים

2. **מסמך תכנון** — `docs/PLAN.md`

   מסמך התכנון מתאר את הארכיטקטורה והתכנון הטכני של הפרויקט, וכולל:

   - תרשימי C4 Model (Context, Container, Component, Code)
   - תרשימי UML לתהליכים מורכבים ותרשימי deployment
   - החלטות ארכיטקטוניות (ADRs) עם רציונל, trade-offs ואלטרנטיבות
   - תיעוד API וממשקים, סכימות נתונים וקונטרקטים

3. **מסמך משימות** — `docs/TODO.md`

   מסמך המשימות מפרט את כל המשימות הנדרשות ליישום הפרויקט:

   - רשימת משימות מפורטת עם עדיפויות וסטטוס (טרם החל / בתהליך / הושלם)
   - חלוקה לשלבים (phases) עם אבני דרך
   - הקצאת אחריות לכל משימה
   - קריטריוני סיום (definition of done) לכל משימה

### 2.3 מסמכי PRD ייעודיים לאלגוריתמים ומנגנונים

**דרישה חשובה:** לכל אלגוריתם ספציפי, מנגנון מרכזי, או רכיב טכני מורכב בפרויקט — יש ליצור מסמך PRD ייעודי ונפרד. לדוגמה:

- `docs/PRD_ml_algorithm.md` — אלגוריתם למידת מכונה
- `docs/PRD_authentication.md` — מנגנון אימות משתמשים
- `docs/PRD_search_engine.md` — מנוע חיפוש
- `docs/PRD_caching.md` — מערכת caching

כל מסמך PRD ייעודי כולל:

- תיאור מפורט של האלגוריתם או המנגנון, כולל רקע תיאורטי
- דרישות ספציפיות, קלט/פלט מצופה, ומדדי ביצוע
- אילוצים ומגבלות, חלופות שנשקלו, והסבר על הבחירה
- קריטריוני הצלחה ותרחישי בדיקה ספציפיים

## 3. תיעוד קוד ומבנה פרויקט

תיעוד קוד נכון ומבנה פרויקט מסודר הם יסודות חיוניים לפיתוח תוכנה מקצועי. תיעוד טוב מאפשר למפתחים אחרים להבין את הקוד במהירות, להשתמש בו בצורה נכונה, ולתרום לפרויקט ביעילות.

### 3.1 מבנה פרויקט מודולרי

ארגון נכון של מבנה הפרויקט הוא מפתח לתחזוקה יעילה ולהתפתחות עתידית של הקוד. עקרונות הארגון כוללים חלוקה לוגית של הפרויקט לתיקיות לפי תפקיד, כגון קוד מקור, בדיקות, תיעוד, נתונים, תוצאות, קונפיגורציה ומשאבים. הארגון יכול להיות מבוסס-תכונות (feature-based) או בארכיטקטורה שכבתית (layered architecture), תוך הפרדה ברורה בין קוד, נתונים, תוצאות ותיעוד.

### 3.2 כלל גודל הקבצים — מקסימום 150 שורות

**כל קובץ קוד לא יעלה על 150 שורות קוד** (שורות ריקות ושורות הערה לא נספרות). כאשר קובץ חורג מהמגבלה, **יש לפצל אותו למספר קבצים** — לעולם לא לדחוס קוד כדי להתאים למגבלה.

אסטרטגיות פיצול:

- **חילוץ פונקציות עזר** — פונקציות עצמאיות לקובץ נפרד
- **חילוץ mixin** — כאשר למחלקה מספר אחריויות
- **פיצול 50/50** — כאשר יש שני חצאים לוגיים (קריאה/כתיבה)
- **חילוץ קבועים** — קבועים לקובץ `constants.py`
- **חילוץ מודלים** — הגדרות מודלים לקובץ נפרד

### 3.3 איכות קוד והערות

איכות הקוד נמדדת לא רק בפונקציונליות שלו אלא גם בקלות הקריאה והתחזוקה שלו. תקני הערות בקוד (Code Comments Standards) [6], [7], [8] דורשים שהערות יסבירו את ה-"למה" ולא רק את ה-"מה". כל פונקציה, מחלקה ומודול צריכים Docstrings מפורטים. ההערות צריכות להסביר החלטות עיצוב מורכבות, לתעד הנחות ותנאים מוקדמים, ולהתעדכן יחד עם שינויי הקוד.

עקרונות כתיבת קוד איכותי כוללים שימוש בשמות משתנים ופונקציות תיאוריים ומדויקים, כתיבת פונקציות קצרות וממוקדות העומדות בעיקרון האחריות היחידה (single-responsibility), הימנעות ממקור כפול לפי עיקרון DRY (Don't Repeat Yourself), ושמירה על עקביות בסגנון לאורך כל הפרויקט.

## 4. ארכיטקטורת SDK ועיצוב מונחה-עצמים

### 4.1 ארכיטקטורה מבוססת SDK

כל פונקציה מרכזית של לוגיקה עסקית חייבת להיות נגישה דרך שכבת SDK. ה-SDK הוא נקודת הכניסה היחידה לכל הצרכנים: תפריטים, GUI, CLI, אינטגרציות עם צדדים שלישיים ושירותים עתידיים.

```
External Consumers (GUI / CLI / REST / Third Party)
        |
        v
   +---------+
   |   SDK   |  <-- Single entry point for ALL logic
   +----+----+
        |
        v
   +-------------+
   |   Domain    |  <-- Services, models, orchestrators
   |  Services   |
   +----+--------+
        |
        v
   +----------------+
   | Infrastructure |  <-- DB, file I/O, external APIs
   +----------------+
```

דרישות ארכיטקטוניות:

- כל פונקציה עסקית חשופה דרך מחלקת SDK
- **אין לוגיקה עסקית** בשכבות GUI, CLI או בקרים — שכבות אלו מאצילות ל-SDK
- צרכנים חיצוניים יכולים לייבא את ה-SDK ולהפעיל את כל הפעולות ללא גישה למודולים פנימיים

### 4.2 עיצוב מונחה-עצמים (OOP) — ללא כפילות קוד

יש לכתוב קוד בגישת OOP. אין לכפול קוד. כאשר אותה לוגיקה מופיעה בשני קבצים או יותר, יש לחלץ אותה למודול משותף, מחלקת בסיס, או mixin.

- אותו גוף פונקציה בשני קבצים או יותר — חילוץ למודול משותף
- אותו דפוס try/except בשלושה קבצים או יותר — יצירת פונקציית עטיפה
- שיטה זהה בשלוש מחלקות או יותר — יצירת מחלקת בסיס או mixin
- לוגיקה מועתקת עם וריאציות קלות — שימוש בתבנית Template Method

כללי Mixin:

- כל mixin מספק דאגה אחת בלבד
- mixins לא דורסים שיטות אחד של השני
- mixins חייבים להיות ניתנים לבדיקה באופן עצמאי

## 5. שומר סף API ובקרת קצב

### 5.1 שומר סף מרכזי (API Gatekeeper)

כל קריאות API החיצוניות חייבות לעבור דרך **שומר סף מרכזי**. שומר הסף מטפל בהגבלת קצב, תורות, ניסיונות חוזרים וניטור.

דרישות שומר הסף:

- **אין קריאות API ישירות** שעוקפות את שומר הסף
- הגבלות קצב נאכפות לפני כל קריאה
- גלישה מועברת לתור (queue), לא נדחית
- כל קריאות API נרשמות לניטור

```python
# API Gatekeeper Interface

class ApiGatekeeper:
    """Centralized API call manager."""

    def __init__(self, config: RateLimitConfig):
        """Initialize with rate limit config."""
        ...

    def execute(self, api_call, *args, **kwargs):
        """Execute API call through gatekeeper.
        - Check rate limits before execution
        - Queue if limit reached
        - Retry on transient failures
        - Log all calls
        """
        ...

    def get_queue_status(self) -> QueueStatus:
        """Return queue depth and stats."""
        ...
```

### 5.2 קונפיגורציית הגבלת קצב

**כל הגבלות הקצב חייבות להיקרא מקובץ קונפיגורציה, לעולם לא לקודד בקוד.**

```json
// Rate Limit Configuration
{
  "rate_limits": {
    "version": "1.00",
    "services": {
      "default": {
        "requests_per_minute": 30,
        "requests_per_hour": 500,
        "concurrent_max": 5,
        "retry_after_seconds": 30,
        "max_retries": 3
      }
    }
  }
}
```

### 5.3 ניהול תורות לגלישה

כאשר הגבלות הקצב מגיעות לסף, שומר הסף **חייב** להעביר בקשות לתור במקום לדחות או לקרוס:

- תור FIFO לבקשות ממתינות
- עומק תור מקסימלי מוגדר בקונפיגורציה
- התראת לחץ (backpressure) כאשר התור מלא
- מנגנון ריקון שמעבד בקשות כאשר חלונות הקצב מתאפסים

## 6. פיתוח מונחה-בדיקות ואבטחת איכות

### 6.1 תהליך TDD — אדום, ירוק, שיפור

כל הפיתוח חייב לעקוב אחר Test-Driven Development: אדום (RED) — ירוק (GREEN) — שיפור (REFACTOR).

- כל מודול חייב לכלול קובץ בדיקות מתאים
- כל פונקציה/שיטה ציבורית חייבת לכלול לפחות בדיקה אחת
- הבדיקות מכסות הן את המסלול התקין והן מקרי שגיאה
- בדיקות נכתבות **לפני** או במקביל ליישום, לא כמחשבה שלאחר מעשה

מבנה בדיקות:

```
tests/
  unit/
    test_<module>/         # Mirror src/ structure
      test_<file>.py
  integration/
    test_<feature>.py
  conftest.py              # Shared fixtures
```

כללי בדיקות:

1. כל מודול חדש חייב קובץ בדיקות מתאים
2. כל פונקציה ציבורית חייבת לפחות בדיקה אחת
3. בדיקות מסלול תקין **וגם** מקרי שגיאה
4. שימוש ב-fixtures מ-`conftest.py` לנתוני בדיקה משותפים
5. Mock לתלויות חיצוניות (מסד נתונים, קבצים, API)
6. קבצי בדיקות עומדים גם הם בכלל 150 השורות
7. אין בדיקות שתלויות בשירותים חיצוניים

### 6.2 כיסוי בדיקות — מינימום 85%

**כיסוי בדיקות גלובלי חייב להיות 85% ומעלה.** חבילת הבדיקות חייבת להיכשל אם הכיסוי יורד מתחת לסף זה.

קונפיגורציה ב-`pyproject.toml`:

```toml
# Coverage Configuration
[tool.coverage.run]
source = ["src"]
omit = ["src/main.py", "*/tests/*", "src/**/gui/*"]

[tool.coverage.report]
fail_under = 85
```

סוגי הכיסוי הנדרשים כוללים כיסוי הצהרות (Statement coverage), כיסוי ענפים (Branch coverage), וכיסוי מסלולים (Path coverage) למסלולים קריטיים.

### 6.3 טיפול במקרי קיצון ותקלות

זיהוי ותיעוד מקרי קיצון (Edge Cases) הוא חלק חיוני מפיתוח תוכנה איכותי. יש לזהות תנאי גבול, לתעד כל מקרה עם תיאור מפורט, ולכלול צילומי מסך של תקלות כאשר רלוונטי. מנגנוני טיפול בשגיאות חייבים לכלול תכנות הגנתי, הודעות שגיאה ברורות, רישום לוגים מפורט, והידרדרות חלקה (graceful degradation).

### 6.4 תוצאות בדיקה צפויות

יש לתעד את תוצאות ההרצה הצפויות לכל בדיקה, לייצר דוחות automated testing עם pass/fail rates, ולשמור לוגים של הרצות מוצלחות וכושלות.

## 9. מחקר וניתוח תוצאות

מחקר מעמיק וניתוח תוצאות הם מה שמבדיל בין פרויקט תוכנה רגיל לבין עבודה מקצועית ברמת מצוינות. החלק המחקרי כולל ניסויים שיטתיים, ניתוח כמותי ואיכותי, והצגה ויזואלית של התוצאות.

### 9.1 חקר פרמטרים

ניתוח רגישות פרמטרים (Sensitivity Analysis) הוא תהליך שיטתי של בדיקת השפעת פרמטרים שונים על ביצועי המערכת. התהליך כולל ביצוע ניסויים שיטתיים עם שינוי מבוקר של פרמטרים, תיעוד מדויק של השפעת כל פרמטר, ושימוש בשיטות ניתוח מתקדמות כגון נגזרות חלקיות, ניתוח מבוסס שונות, או גישת "פרמטר-אחד-בכל-פעם" (OAT).

### 9.2 מחברת ניתוח תוצאות

מחברת ניתוח תוצאות (Results Analysis Notebook) היא כלי מרכזי להצגת המחקר. עומק הניתוח מושג באמצעות Jupyter Notebook או כלים דומים, ביצוע ניתוח מתודי של תוצאות הניסויים, השוואה בין אלגוריתמים, תצורות או גישות שונות, והכללת הוכחות מתמטיות או ניתוחים תיאורטיים. יש להשתמש ב-LaTeX לכתיבת משוואות ונוסחאות מקצועיות, ולכלול אסמכתאות לספרות אקדמית.

### 9.3 הצגה ויזואלית של תוצאות

ויזואליזציה איכותית של נתונים חיונית להעברת המסר המחקרי. סוגי הויזואליזציות כוללים Bar charts להשוואות, Line charts למגמות, Scatter plots למתאמים, Heatmaps לרגישות פרמטרים, Box plots להתפלגויות, ו-Waterfall charts לניתוח שינויים. איכות הגרפים נמדדת בבהירות התוויות, שימוש בצבעים עקביים ונגישים, כיתובים מפורטים ומקרא ברור, ורזולוציה גבוהה.

## 13. תקני איכות בינלאומיים

תקן ISO/IEC 25010 [15] מגדיר מודל מקיף לאיכות תוכנה המכסה שמונה מאפייני איכות עיקריים.

### 13.1 מאפייני איכות מוצר

- **התאמה פונקציונלית** (Functional Suitability) — שלמות, נכונות, והתאמה
- **יעילות בביצועים** (Performance Efficiency) — זמני תגובה, ניצול משאבים, יכולת
- **תאימות** (Compatibility) — יכולת פעולה הדדית ודו-קיום
- **שימושיות** (Usability) — קלות למידה, הפעלה, נגישות והגנה מפני שגיאות
- **אמינות** (Reliability) — בשלות, זמינות, סובלנות לתקלות, ויכולת התאוששות
- **אבטחה** (Security) — סודיות, שלמות, אימות, אחריותיות
- **תחזוקתיות** (Maintainability) — מודולריות, שימוש חוזר, ניתנות לניתוח, לשינוי ולבדיקה
- **ניידות** (Portability) — התאמה, ניתנות להתקנה, וניתנות להחלפה

## 14. ארגון הפרויקט כחבילה

ארגון הקוד כחבילה (package) הוא עיקרון יסוד בפיתוח תוכנה מקצועי. חבילה מאורגנת נכון מאפשרת שימוש חוזר בקוד, ניהול תלויות ברור, הפצה והתקנה פשוטה, ובדיקות מובנות.

### 14.1 קובץ הגדרת החבילה

כל חבילה חייבת לכלול קובץ `pyproject.toml` (מועדף) או `setup.py` המפרט שם, גרסה, תיאור, מחבר, רישיון ותלויות.

### 14.2 קבצי `__init__.py`

קבצי `__init__.py` חייבים להימצא בתיקייה הראשית ובכל תת-תיקייה. מומלץ להשתמש בהם לייצוא ממשקים ציבוריים באמצעות `__all__` ולהגדרת `__version__`.

### 14.3 נתיבים יחסיים

כל הייבואים חייבים להשתמש בנתיבים יחסיים או בשמות חבילות, לעולם לא בנתיבים מוחלטים. גם קריאה/כתיבה של קבצים נעשית ביחסית לנתיב החבילה.

### 14.4 רשימת בדיקה: ארגון כחבילה

1. **קובץ הגדרת חבילה:** האם קיים `pyproject.toml`? האם מכיל שם, גרסה ותלויות? האם התלויות מפורטות עם גרסאות?
2. **קובץ `__init__.py`:** האם קיים בתיקייה הראשית? האם מייצא ממשקים ציבוריים? האם `__version__` מוגדר?
3. **מבנה תיקיות:** האם קוד המקור בתיקייה ייעודית? האם הבדיקות ב-`tests/`? האם התיעוד ב-`docs`?
4. **נתיבים יחסיים:** האם כל הייבואים יחסיים? האם נמנעים נתיבים מוחלטים?

## 15. עיבוד מקבילי וביצועים

שימוש במעבדים מרובים (multiprocessing) ובחוטי ביצוע מרובים (multithreading) חיוני לביצועים אופטימליים של תוכנה מודרנית.

### 15.1 הבדל בין Multiprocessing ל-Multithreading

**Multiprocessing** מתאים לפעולות תובעניות מבחינת מעבד (CPU-bound): חישובים מתמטיים, עיבוד תמונות, אימון מודלים. כל תהליך פועל בזיכרון נפרד ומנצל ליבת מעבד שונה.

**Multithreading** מתאים לפעולות I/O-bound: קריאות רשת, גישה למסדי נתונים, קריאה/כתיבה של קבצים. חוטי ביצוע מאפשרים פעולות אחרות בזמן המתנה.

### 15.2 בטיחות חוטים

בטיחות חוטים (thread safety) קריטית: הגנה על משתנים משותפים באמצעות מנעולים, שימוש ב-`queue.Queue` להעברת מידע, הימנעות מנעילה הדדית, ושימוש ב-context managers.

### 15.3 רשימת בדיקה: עיבוד מקבילי

1. **זיהוי פעולות:** זיהוי פעולות CPU-bound/I/O-bound, בחירת כלי נכון, הערכת תועלת
2. **יישום:** מספר תהליכים/חוטים דינמי, שיתוף נתונים בטוח, מניעת זליגת זיכרון
3. **ניהול משאבים:** סגירה נכונה, טיפול בחריגות, מניעת זליגת זיכרון
4. **בטיחות:** הגנה על משתנים משותפים, מניעת מצבי תחרות ונעילה הדדית

## 16. עיצוב מודולרי ואבני בניין

עיצוב מבוסס אבני בניין (building blocks design) הוא גישה מודולרית בה כל רכיב הוא יחידה עצמאית עם ממשק מוגדר היטב.

### 16.1 מבנה אבן בניה

כל אבן בניה מוגדרת על ידי:

- **נתוני קלט** (Input Data) — סוגי נתונים, תחום תקף, תלויות חיצוניות, ולידציה מקיפה
- **נתוני פלט** (Output Data) — סוגי נתונים, פורמט, התנהגות במקרה קצה
- **נתוני הגדרה** (Setup Data) — פרמטרים עם ברירות מחדל, קונפיגורציה, אתחול

### 16.2 עקרונות עיצוב

**אחריות יחידה** (Single Responsibility) — כל אבן בניה אחראית למשימה אחת. **הפרדת דאגות** (Separation of Concerns) — כל אבן בניה עוסקת בהיבט אחד. **שימוש חוזר** (Reusability) — אבני בניה עצמאיות ולא תלויות בקוד ספציפי. **יכולת בדיקה** (Testability) — כל אבן בניה ניתנת לבדיקה באמצעות הזרקת תלות.

### 16.3 דוגמה לאבן בניה

```python
# Building Block Example

class DataProcessor:
    """
    Input:  raw_data (List[Dict]),
            filter_criteria (Dict)
    Output: processed_data (List[Dict])
    Setup:  processing_mode ('fast'/'accurate'),
            batch_size (int, default: 100)
    """

    def __init__(self, processing_mode='fast',
                 batch_size=100):
        self.processing_mode = processing_mode
        self.batch_size = batch_size
        self._validate_config()

    def process(self, raw_data, filter_criteria):
        self._validate_input(raw_data,
                             filter_criteria)
        return self._do_processing(
            raw_data, filter_criteria)

    def _validate_config(self):
        if self.processing_mode \
                not in ['fast', 'accurate']:
            raise ValueError("Invalid mode")
        if self.batch_size <= 0:
            raise ValueError("Batch size > 0")

    def _validate_input(self, data, criteria):
        if not isinstance(data, list):
            raise TypeError("data must be list")
        if not isinstance(criteria, dict):
            raise TypeError("criteria must be dict")
```

1. **תיעוד:** PRD, ארכיטקטורה, README, תיעוד API, ספר פרומפטים
2. **קוד:** מבנה מודולרי, קבצים ≤ 150 שורות, הערות ו-docstrings, עקביות סגנון
3. **קונפיגורציה:** קבצים נפרדים, `.env-example`, ללא סודות, `.gitignore`
4. **בדיקות:** כיסוי 85%+, מקרי קיצון, טיפול בשגיאות, מחברת ניתוח, דוחות אוטומטיים
5. **מחקר:** חקר פרמטרים, ניתוח רגישות, מחברת ניתוח, גרפים
6. **ויזואליזציה:** גרפים איכותיים, צילומי מסך, תרשימי ארכיטקטורה
7. **עלויות:** טבלת tokens, ניתוח מפורט, אופטימיזציה
8. **הרחבה:** נקודות הרחבה, דוגמאות תוספים, ממשקים
9. **כלליים:** Git history, רישיון, ייחוס, deployment
